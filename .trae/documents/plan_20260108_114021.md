# 图片处理重构方案

## 1. 重构方案概述

### 1.1 核心思路
- 彻底移除 Base64 图片处理方式
- 统一使用 Supabase Storage 存储图片
- 数据库只存储图片路径（字符串或字符串数组）
- 前端直传 Storage，无需后端介入

### 1.2 目录结构调整
- 创建 `lib/imageHandler.ts`：集中管理图片上传、压缩和URL生成逻辑
- 更新 `lib/yearlySummaryApi.ts`：适配新的 `yearly_images` 表
- 更新 `lib/diaryApi.ts`：适配新的 `image_paths` 字段

## 2. 详细实现计划

### 2.1 创建图片处理工具（lib/imageHandler.ts）
- 实现图片压缩功能
- 实现单个图片上传到 Storage
- 实现多个图片批量上传
- 实现从 Storage 生成访问 URL

### 2.2 重构年度总结图片处理
- 更新 `yearlySummaryApi.ts` 中的图片相关函数
- 修改 `yearly-summary.tsx` 中的上传和渲染逻辑
- 适配新的 `yearly_images` 表结构

### 2.3 重构日记图片处理
- 更新 `diaryApi.ts` 中的图片相关函数
- 修改日记编辑和详情页面的图片处理逻辑
- 适配新的 `image_paths` 字段（jsonb 数组）

### 2.4 清理旧代码
- 移除所有 Base64 相关的代码
- 删除不再使用的图片处理函数

## 3. 技术实现细节

### 3.1 图片上传流程
1. 前端选择图片
2. 前端进行压缩处理
3. 生成唯一文件名（按指定格式）
4. 上传到对应的 Supabase Storage Bucket
5. 获取 storage path 并写入数据库

### 3.2 图片加载流程
1. 从数据库读取图片路径（单图或数组）
2. 通过 Supabase Storage API 生成访问 URL
3. 前端渲染 URL

### 3.3 路径格式
- 年度总结图片：`yearly/{index}.webp`
- 日记图片：`{year}/{yyyymmdd}_{index}.webp`

## 5. 注意事项与常见错误

### 5.1 常见错误
- 忘记压缩图片导致上传速度慢
- 重复上传相同图片
- 路径格式错误导致文件找不到
- 权限配置错误导致无法访问图片

### 5.2 最佳实践
- 始终进行图片压缩
- 使用统一的命名规范
- 错误处理和重试机制
- 图片上传进度反馈
